# orc — 改修タスクのオーケストレーション

意図が固まった改修タスクを最小確認で自走させるためのスキル。implementer に委任 → code-reviewer で点検 → 完了報告、の 2 層構成。

新規ゲーム制作は `newgame` スキル（クリエイティブ専用、一次資料・発散・美学判断・実機検証の固有工程あり）を使う。本スキルは「意図が固まった改修」専用で、両者は軸が違う。

## 動作フロー

### 1. 最小確認（基本は聞かない）

**原則: 依頼文から「何を変えるか / 対象 / 成功条件」を自分で埋めて即着手する。** ユーザーは SE であり、実装の考えごとは orc 側に委ねている。聞くこと自体がユーザーの負担なので、デフォルトは「聞かない」。

確認するのは、次の両方を満たす分岐が **実在するときだけ**:

1. どちらを選ぶかで **成果物が実際に変わる**（聞かずに進めると作り直しになる）
2. orc 側の情報（コード・既存慣習・依頼文脈）からは **合理的に決められない**

該当しなければ確認なしで委任する。「対象がどのファイルか」「成功条件をどう確認するか」も、コードを読めば分かる範囲は自分で確定する（必要なら委任前に Explore / Grep で裏取りする）。

**確認しないこと**（orc / implementer が決める）: 設計細部、実装方針、ライブラリ選定、命名規則、内部構造、テスト方針、そして「複数の実装案のどれを採るか」。これらは「実装相談」であり、ユーザーに振らずこちら側で決める。迷ったら既存コードの慣習に最も寄る案を採り、選んだ理由を完了報告に 1 行残す。

**確認するときの作法**（負担最小化）:

- `AskUserQuestion` を **1 回だけ**。複数論点があっても 1 回のツール呼び出しにまとめる
- 推奨案を **先頭** に置き、ラベル末尾に「(推奨)」を付ける。ユーザーが頷くだけで進む状態にする
- 選択肢は具体的で、そのまま選べる粒度にする（「どうしますか？」のような開放質問にしない）
- 「念のため」「確認だけ」の質問はしない。聞く＝結果が変わる、を満たすときだけ

### 2. 委任

確認が不要なら即、確認したならユーザー回答を得たら → `Task` ツールで `subagent_type: implementer` 起動。

`prompt` には「何を変えるか / 対象 / 成功条件」を転記する。orc 側で確定・補完した設計判断があればそれも渡してよいが、不要な予断は書かない。

委任後、ユーザーへの中間報告はしない。implementer が完了を返すまで待つ。

### 3. レビュー（必須）

implementer 完了 → 続けて `Task` ツールで `subagent_type: code-reviewer` を起動する。レビュー観点は `code-reviewer.md` に定義済みなので、orc 側は対象差分の指定だけ渡す。

**レビュー結果による分岐**:

- **OK のみ**: そのまま完了報告へ
- **軽微な提案**: 完了報告に併記し、対応するかをユーザーに 1 行で確認（`feedback_review_minor_proposals.md` と整合）
- **修正必須あり**: implementer に再委任（指摘内容を prompt に転記）。再委任は **1 往復まで**。2 回目の修正必須が出たらユーザーに状況を報告し、判断を仰ぐ

### 4. 完了報告

以下のフォーマットでユーザーに返す。

- **変更ファイル**: リポジトリ相対パスで列挙
- **動作確認**: 実施した確認内容と結果（テスト / 起動 / 目視のいずれか）
- **レビュー**: code-reviewer の結論サマリ（OK / 軽微提案あり / 修正必須あり → 対応済み）
- **commit**: hash と 1 行メッセージ（複数あれば全部）
- **意図との差分**: 委任時の意図と異なる対応があれば明記。なければ「なし」
- **要するに**: 上記の詳細とは別に、報告の **末尾に必ず 1〜2 行** で「結局どうなったか」を専門用語を避けて書く。技術ログを追わなくても結論が分かる平易な言葉にする（例: 「ダッシュボードの時計が数秒ずれていたのを直した。次回起動から正しく出る。今のところ追加対応は不要」）。詳細セクションの繰り返しではなく、ユーザーにとっての意味で締める

詳細フォーマットはこれまでどおり。「要するに」を 1 行足すのが今回の追加点。

## 事前にユーザー確認が必須の例外操作

以下を implementer が行う必要が出た場合、orc は委任を一時停止してユーザーに承認を求める。

- `git push` / タグ push / PR 作成
- 本番デプロイコマンド
- 外部 API 課金が発生する操作
- 既存データの削除（DB レコード、ファイル削除、`rm -rf` 系）

これら以外は確認不要。

## agent team との関係

orc 内では agent team は **使わない**。理由:

- 実装タスクは議論より実行が主眼。teammate 同士の反証が本質的に不要
- agent team の真価は発散・反証ラウンド（仕様検討 / 設計レビュー）にあり、用途が違う
- star に潰れたら subagent と同じだが、コストは桁で高い

implementer が「設計判断が複数案あって決められない」と報告してきた場合、まず **orc 側で決める**（実装相談はユーザーに振らない方針）。既存コードの慣習・依頼文脈から最も妥当な案を選び、選択理由を完了報告に残して再委任する。

例外的にユーザーへ返すのは、その判断が「事前確認必須の例外操作」（push / 削除 / 課金 / 互換破壊・データモデル変更）に踏み込む場合のみ。純粋な実装方針の分岐でユーザーに状況を投げ返さない。agent team を起こすかどうかも、その分岐がユーザー領域に達したときに限ってユーザー判断とする。
