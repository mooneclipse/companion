> **ステータス**: 設計確定版 (2026-07-08、チケット #48)。game-designer 3 並列 (敵・遭遇 / ラン構造 / ビルド深化のレーン固定) → game-critic 反証 → lead 統合、の工程を完了。**コードは未変更** (この文書は設計のみ)。
> **実装 GO はオーナーレビュー後**。実装は ver2.0 → 2.1 → 2.2 の 3 増分に分割し、各増分ごとに playtester 実機検証 + bot 勝率計測を通す。
> 旧版 (2026-06-10 AI 叩き台) は本改稿で置き換えた。叩き台の敵アーキタイプ 5 種は §3.2 の採否表に吸収。

## 1. 前提 — 何を直し、何を変えないか

**感想 7/10 の課題 (優先順)**:
1. リプレイ性: 敵が 1 パターンしかなく何度も遊べない (ver2 最優先)
2. カードの分かりにくさ: 効果が不明瞭なカードがいくつかあった
3. 初接触 UX: v1.1〜1.3 で修正済みだが、ver2 では**新概念を増やしすぎないこと自体が最良の対策** (critic 反証の結論)

**変えない (当たり資産)**:
- 核メカ「LIGHT=リソース」(明攻×明るさ / 闇攻×暗さ / 被ダメ×暗さ / 撃つと暗くなる) と明度スケール 3 関数
- 「気を抜くと負ける」難易度感 = bot 勝率帯 30〜55% (v1 の確定値が基準点)
- 既存 6 体の敵・既存 13 カードの数値 (**1 数値も変えない**。7/10 の基準点を動かさない)
- 決定論 intent ループ (動的 AI 禁止)、全 DOM・はみ出し 0・svh 対応、純静的 PWA、タップのみ操作

**コード実査で確定した設計上の重要事実** (designer 報告を lead が一次確認済み):
- LIGHT は毎戦 50 リセット (`app.js:193`)。戦闘間の持ち越しは存在しない
- 集光 `nextAtkMult` は**ターン開始で毎回 1 にリセット** (`app.js:240`) — 文言「次の攻撃 ×2」と実挙動が乖離 = 課題 2 の最有力犯
- 敵 block は減衰せず蓄積 (`app.js:441`)。Guard 型敵の数値は膠着リスクと隣り合わせ
- 報酬カードはタップ即取得 (`app.js:574-577`) — 初見カードを理解前に取ってしまう構造
- Math.random はあかりでは仕様で許可済み (シャッフル・報酬抽選、`app.js:183`)

**ver2.0 を敵+可読性+音に絞る根拠** (critic 反証の採用):
- 3 レーン案の ver2.0 推奨を全部足すと新概念 10 個超・v1 本体 1938 行の 6〜7 割の差分 = 課題 3 (初接触 UX) の自傷再発。あそびかた overlay に収まらない
- Drain は**既存の**影撃ち・蝕 (闇型カード) を初めて立たせる設計。新カード 6 を同時投入すると「光型一強が敵で崩れたのかカードで崩れたのか」計測不能になる。1 変数ずつ動かし勝率とビルド分布を見てから次を足す
- 音は新概念ゼロ・説明不要で「敵以外の面白さ」を足せる唯一の低リスク増分 (mineroad で実装パターン確立済み)

## 3. ver2.0 詳細設計

### 3.1 遭遇構造: floor=敵固定 → ティア固定プール抽選

```
weak   (floor1-2, 重複なしで2抽選): 火の粉, 蛾, 灯喰い
mid    (floor3-4, 重複なしで2抽選): 影, 二つ火, 埋み火
strong (floor5, 1抽選):            篝, 常闇, 灯呑み
boss   (floor6, 固定):             大いなる闇
```

- `startFloor` の `ENEMIES[floor-1]` 直引き (`app.js:204`) を、ラン開始時に一括生成した敵列 `G.encounters[floor-1]` 参照へ差し替え
- **ランタイム HP/intent スケーリングはしない**: プールがティアに完全ロックされている以上、各敵はその階専用の固定値 1 本で足りる。スケール係数は enemies.js の verbatim 性 (書いてある数値 = 画面の数値) とテスト決定論を汚すだけ (叩き台からの意図的逸脱)
- 体感バリエーションの正直な見積もり (critic 指摘採用): 組合せ総数は 108 だが weak 帯は 2〜3 ターンで終わるため、**体感の本体は mid の顔ぶれ × strong の壁 ≒ 9 通り前後**。「最低数回は新鮮」は満たすが、終着 (ボス) が 1 つな限り収束は残る — ボス 2 体目 (ver2.2) までは解けない構造と認識しておく

### 3.2 叩き台 (2026-06-10) 5 アーキタイプの採否

| 叩き台 | 判定 | 理由 |
|---|---|---|
| A 闇喰らい (弱A+毎ターンDim) | **修正採用 → Drain 型「灯喰い」に変換 (ver2.0)** | 毎ターン Dim は被ダメ増と二重罰で対抗が「灯を積むだけ」の一方通行。Drain なら「先に撃って暗くなっておけば喰われる量が減る」逆説の駆け引きが立つ |
| B 鏡像 (明るいほど強く殴る) | **採用・ver2.1 送り (写し火)** | LIGHT 連動のライブ intent 表示は初接触 UX と直結する唯一の表示リスク。抽選構造の手応えを見てから載せる |
| C 守りの炎 | **採用 (ver2.0、埋み火)** | 新 primitive 不要。ただし敵 block 蓄積仕様 (`app.js:441`) により高頻度 Guard は膠着製造機 — 供給量を中階層期待 DPT の 4〜5 割に絞り「集光バーストで貫通」を正解に残す |
| D 群れ (3体+死亡強化) | **ver2.1 送り** | onAllyDeath primitive + 3 体視認性検証でコスト最大。二つ火が多体戦の座を当面担う |
| E 蝕む闇 (Charge2連+暗いほど痛い特大) | **修正採用 (ver2.0、常闇)・専用 primitive 削除** | 「暗いほど痛い」は takenMult が既にやっている (L0 で ×1.45)。素の特大数値 + Charge 2 連で成立、新規実装ゼロ |

### 3.3 新 intent primitive: ver2.0 は **Drain の 1 個だけ**

```
{type:"Drain", n}  — stolen = min(G.light, n); G.light -= stolen; e.block += stolen
```

- enemyAct に 1 case (既存 Dim + 敵 block 加算の合成) + cue「あかり -N」(既存 Dim cue と同型・白文字黒縁) + **intentIcon への Drain case 追加** (critic 指摘: これが無いと「予告を見て対応」が成立しない。見積もりに含める)
- 面白さの中身: ①灯ホーディングに初めてリスクが付く ②「Drain 予告を見て先に攻カードで灯を捨てて痩せさせる」対抗手が**現有カードだけ**で打てる ③奪った光が敵の鎧になる = 資源の移動が画面で見える ④燭台 (毎ターン灯+8) が Drain 相手には給餌になる両刃化 = 既存カードの評価が相手で変わる
- **既知リスク (検証ゲートで測る)**: 喰われる→暗くなる→takenMult 増→敵硬化、の三重連鎖は v1 が TAKEN 0.6→0.45 で潰したスパイラルの敵側からの再導入。光型デッキ (闇カード非所持) で灯呑み戦が詰みに近づかないか bot 実測必須

### 3.4 新敵 4 体 (初期値。playtester 実測で調整前提)

```js

// 「教師役」は名乗らない — Drain の本番は strong 帯の灯呑み
{ id:"higui", name:"灯喰い", hp:20, units:1, boss:false,
  intent: [[{type:"Drain", n:8}], [{type:"A", n:6}]] },

// 埋み火 (mid帯)。「拒む」型。block 供給 13/3ターン ≒ 中階層期待DPT(8〜12)の4〜5割
{ id:"uzumibi", name:"埋み火", hp:28, units:1, boss:false,
  intent: [[{type:"Guard", n:8}], [{type:"A", n:6}, {type:"Guard", n:5}], [{type:"A", n:13}]] },

// 常闇 (strong帯)。「数える」型: 3手の猶予で hp32 を割り切るか、灯+block で耐えるか
// ⚠ A26 は L50 実効 32・L0 実効 38。v1 が篝 18→14 に緩和して勝率帯を作った経緯に
// 逆行する危険値 (critic 指摘)。バランス計測で死因 5 割超なら実効 20 前後 (A:16〜18) へ
{ id:"tokoyami", name:"常闇", hp:32, units:1, boss:false,
  intent: [[{type:"A", n:7}], [{type:"Charge"}], [{type:"Charge"}], [{type:"A", n:26}]] },

// 灯呑み (strong帯)。「喰う」型の完成形: 灯を細く保つ低空飛行 (闇型が刺さる) か、
// 喰われる前提の速攻か
{ id:"hinomi", name:"灯呑み", hp:36, units:1, boss:false,
  intent: [[{type:"Drain", n:12}], [{type:"A", n:9}], [{type:"A", n:9}]] },
```

- 名前の文体: 現行の和語 (火の粉/蛾/影/篝) に揃える。カード名 (蝕・残光・燭台) と衝突なし
- **strong プールの分散ガード** (critic「壁の高さの抽選」指摘への設計解): strong 3 種は「速攻を強いる (常闇)/長期戦を強いる (灯呑み)/リズムを読む (篝)」と**問いの種類**を変える。bot 計測で 3 種の勝率が大きく割れたら、緩和は「弱い側を上げる」でなく「強い側の数値を下げる」方向 (勝率帯 30〜55% の維持が正)

### 3.5 seed 基盤 (決定論テストの回復)

- ラン開始 (`newRun`) 時に `G.seed` を 1 個確定 (既定 = crypto.getRandomValues)。`?seed=N` URL パラメータで固定可
- **G.rng (mulberry32) の適用範囲は「ラン開始時の一括事前抽選」のみ**: 敵列 (floor1..5 の id 列) + 報酬候補列 (3 枚 × 5 回分)。**山札 shuffle は従来 Math.random のまま** — shuffle を seed に含めるとプレイ行動が後続抽選を変え「seed=N → 敵列一致」の gate の意味が変質する (critic 指摘の仕様曖昧性をこちら側に確定)
- テスト専用フック `?force=<id5つ>` (floor1..5 の敵 id をカンマ区切り、例 `?force=higui,uzumibi,kage,futatsubi,tokoyami`。不正 id は当該 floor のみ通常抽選へフォールバックし console.warn) を用意。抽選テーブルの**入口**を差し替えるだけで state 直代入ではない (mineroad の「状態注入が創発バグを見逃す」教訓に抵触しない)。本番コードに残置するが URL を知らなければ踏めない・個人利用 PWA なので許容、と仕様として明記
- debug-akari.mjs の更新: 「seed 固定 → 敵列 id 完全一致」gate + `?force=` で新敵 4 体を 1 体ずつ実プレイ経路 (カードタップ→ターン終了) で通す挙動 gate (Drain の min クランプ / 敵 block 蓄積 / Charge 2 連表示)

### 3.6 カード可読性の根治 (課題 2)

**名指しの問題と対処** (悪い順):
1. 集光「次の攻撃 ×2」— 実装はターン内限り (`app.js:240`)。→ 文言「このターン攻×2」+ バフ中 HUD に「×2」バッジ
2. 灯の盾「ブロック=明/8」— 割り算の暗算を強制。→ 実効値ライブ表示で解決 (下記)
3. 打つ/焔/大焔「暗 -6」— 灯が減るのか「暗」が増えるのか語義反転。→ **LIGHT 減は全カード「灯-N」表記に統一** (「暗」は闇スケールの形容のみ)
4. 燭台「毎ターン 灯+8」— 戦闘限りが読めない。→ 「戦闘中 毎ターン灯+8」へ (字数上限内に収める)
5. 影撃ち/蝕/打つ系 — 表示は基礎値で実ダメージ (暗さ・明るさ依存) が見えない。→ 実効値ライブ表示

**実効値ライブ表示** (旗艦。build 案採用): `renderHand` は状態変化のたび全再構築 (`app.js:669`) なので、`buildCardEl` で現在 LIGHT を明度スケール関数に通した**実際の数値**を表示する。打つ →「ダメージ 7▲／灯-6」(▲=明るいほど伸びる / ▼=暗いほど伸びる)。灯を灯すと手札の数字がその場で上がる = 核メカが手札に可視化され、分かりにくさの根と教育を同時に解決 (StS の実値表示と同型)。
- リスク (critic 指摘、playtester gate 化): ▲▼ は新しい記号言語 = 初見説明が 1 個増える。「さっき 7 だったのに 6 になった」初見混乱の逆リスク。→ あそびかたに 1 行「カードの数字は いまの明るさでの値」+ 初見 gate で確認

**報酬画面の 2 タップ確定**: 現行はタップ即取得 (`app.js:574-577`)。1 タップ目 = 選択 + 詳細 2 行 (持続範囲・スケール説明) 表示、2 タップ目で確定。初見カードを理解前に取る構造を塞ぐ。

**あそびかた追記は 2 行以内**: 「あかりを喰う敵がいる」「カードの数字は いまの明るさでの値」。これ以上増える設計は ver2.0 に入れない。

### 3.7 音 (Kenney Game Assets 1, CC0 — ダウンロード済み zip から)

あかりは現状**完全無音**。ライセンス判定は mineroad リスキン時に確定済み (Kenney=CC0 採用 / Solaria=AI 禁止条項で除外 / Tiny Tales・Premium Platformer=不使用)。実装パターンも mineroad で確立済み (`playSfx` cloneNode 使い捨て再生・BGM はユーザー操作起点 + mute トグル)。

- **SFX** (実在確認済みフォルダから。最終選定は実装時に試聴):
  - カードプレイ (攻): `Audio/RPG sounds/chop.ogg` or `knifeSlice.ogg`
  - カードプレイ (灯/守/技): `cloth*.ogg` / `metalClick.ogg` / `bookFlip*.ogg` 系
  - 被ダメ / Drain: `RPG sounds` 打撃系 + Dim/Drain は同系流用
  - UI (ターン終了・報酬選択): `Audio/UI sounds` 系
  - 勝利 / 敗北ジングル: `Audio/Jingle sounds/jingles_NES*.ogg` or `jingles_STEEL*.ogg` (mineroad と同系)
- **BGM**: `Audio/Music loops` の暗め候補 (Sad Descent / Retro Mystic / Flowing Rocks)。ただしいずれもチップチューン寄りで、あかりの静かな闇の美学に合うかは**実機試聴で判断、合わなければ ver2.0 は SFX のみ** (無理に入れない)。
- **画像は不使用** (今回の調査で確定): Game Icons 790 個は UI・入力デバイス系のみでカード意匠に合う絵柄なし、Generic Items は現代日用品、Roguelike 系 16px ピクセルキャラは「闇のシルエット+光輪」の抽象美学と衝突。**あかりの視覚は現行の抽象描画が当たり資産であり、ピクセルアートを混ぜない**
- 配信境界: `server/app.py` STATIC は明示 allowlist 方式のため、音源ファイル分の純追加が必要 (mineroad の 14 本追加と同形)。既存 URL 不変

### 3.9 ver2.0 検証ゲート (playtester)

1. bot 勝率帯 30〜55% 維持 (seed 1..20 の 20 ラン) + **敵別死因分布** (特定新敵が死因 5 割超 = 数値見直しシグナル。特に常闇 A26)。注: seed が固定するのは敵列・報酬列のみで山札 shuffle は非 seed (§3.5) のため、**勝率はラン単位では再現しない** — 帯として評価する
2. strong 3 種それぞれの勝率分散 — 「壁の高さの抽選」化していないか (§3.4)
3. Drain スパイラル: 光型デッキ (闇カード非所持) の灯呑み戦勝率が他 strong と大きく乖離しないか
4. 埋み火の膠着: 敵 block 蓄積で 1 戦のターン数が伸びすぎないか
5. seed 決定論: 同 seed → 敵列・報酬列 verbatim 一致 / `?force=` で新敵 4 体の挙動 gate (実プレイ経路)
6. 実効値ライブ表示の初見 gate: ▲▼ と数字変動の理解 / 2 タップ確定の導線
7. 既存回帰: 全画面はみ出し 0 (412×915 + 短高 412×680/730)・既存 6 作 URL 不変・pageerror 0・SFX 連打 stall なし (cloneNode)

## 4. ver2.1 設計骨子 (敵抽選の実測後に着手)

**着手条件**: ver2.0 のバランス計測でビルド分布 (光型/闇型の選好) と勝率帯が確定していること。

- **新カード 6** (build 案から、critic の修正条件反映):
  - そのまま採用候補: 灯心 (灯+10 1枚引く)・影の帳 (ブロック=暗/8、`blockFromDark` 1 行)・残り火 (闇ダメ7 灯+10)・明滅 (明暗反転、`lightFlip` 1 行・初クリア解禁)

## 6. 横断の設計原則 (critic 反証から昇格)

1. **回復ノブは同時に増やさない**: v1 は回復 2 ノブ (WIN_HEAL/SKIP_HEAL) で勝率帯を 1 回で決めた。泉/休む/祠を重ねると「初回調整 1 回で確定」が数学的に守れない。増分ごとに回復ノブは高々 1 個
2. **核メカの代償を消す報酬を作らない**: 月影型 (暗さペナルティ無効) は当たりの明度綱引きの無効化。軽減 (連続量) までに留める
3. **ランダム性の追加は 1 増分 1 系統**: 敵抽選 (2.0) → 報酬/遺灯 (2.1) → 分岐 (2.2)。同時に足すと不具合とバランスの原因切り分けが死ぬ
4. **overlay 連鎖は最大 2 枚**: 勝利後に 遺灯→報酬→分岐 の 3 連続 overlay はテンポと認知負荷の山 (5〜8 分・片手の看板を守る)
5. **イベント断章は 1 行上限**: ひらがな短句が膨らみ始めたら詩への回帰シグナル (3 作凡庸着地の轍)

## 7. 工程メモ

- 実装は `/newgame` でなく orc (改修) 系統。designer/critic の再起用は不要 (本設計で確定)、implementer → playtester → code-reviewer の直列
- VERSION: v1.3.0 → v2.0.0 (ver2.0 実装時)
- akari/web は静的配信 = ディスク編集で本番即反映のため、**実装・検証は必ず別ポートのコピーで行い、本番ディレクトリは検証完了まで触らない** (v1.1.0 検証時の本番 kill 事故の教訓 + 静的配信の即反映特性)
- 音源追加時は `server/app.py` STATIC allowlist への純追加 + restart が必要 (静的ファイル追加だけでは配信されない)
