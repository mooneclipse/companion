## 0. このドキュメントの位置づけ

これは **Linux 機（常時起動）で稼働する Claude Code への指示文**である。

- 実装・詳細計画・既存環境調査・技術選定は **Linux 機側で行う**。このドキュメントは**要件レベル**にとどめる。
- Windows 側（作成元セッション）で OWNER と対話して確定した仕様を「前提」として渡す。§3 は変更不可の合意事項。§7 は Linux 機が調査・判断する領域。
- 進め方は §8 に従うこと（いきなり実装せず、要件→実装計画→OWNER の承認→実装）。

## 1. 目的（なぜ作るか）

OWNER の**生活の掌握**。具体的には、平日の**帰宅後〜就寝までの自由時間**を、本当にやりたいこと（創作活動、原神のイベント消化など）に向けられるよう**日次で伴走コーチング**する。

重要な性格づけ:

- これは**生産性の取り締まりではない**。対象時間は OWNER の唯一の自由時間であり、原神も創作も「やりたい娯楽」。
- システムの役割は監督官ではなく、**「今夜やりたいって言ってたこと、手をつけられた？」と気にかけてくれる味方**。
- 監視で自由時間を窮屈にすると逆効果。**超低圧**であること、「今日は休む」で**即座に引き下がる**ことが最優先の設計原則。

## 2. コーチングの骨格（意図→行動→振り返りのループ）

```
①意図（何をしたいか）→ ②行動（実際に何をしたか）→ ③振り返り（差を見て次に活かす）
```

- ②行動 = デバイスの活動記録から自動取得（後述）。
- ①意図・③振り返り = 夜ブロックの対話で毎晩まとめて回す。
- OWNER は既存スキル `/want`（やりたいこと登録）・`/retro` を持つ。①③でこれらを流用できる可能性がある（Linux 機で確認）。

## 3. 確定要件（会話で合意済み・変更不可の前提）

### 3-1. 対象スコープ

- **管理対象時間**: 平日 **19:00〜24:00（帰宅〜就寝）の約5時間ブロック**。
  - 実働は夕飯・風呂を除くと **3時間前後**。詰め込まない前提で設計する。
  - この時間帯**以外**（勤務中・睡眠中）は対象外。
- **対象デバイス**: Windows PC ＋ Android スマホ の活動。

### 3-2. 管理リズム — 「夜ブロック × B案（号令＋中間1回）」

1. **号令（ブロック開始）**: 19時台に「今夜なにやりたい？」と意図を1つ決める。
   - 帰宅直後に押し付けない（飯・風呂で落ち着く時間を許容）。OWNER 側から「今夜始める」と振ってトリガーもできること。
2. **中間チェック（多くて1回）**: 21〜22時頃、**宣言した意図から逸れている時だけ** 1回そっと促す。
   - 何度も口を出さない。**すぐ流せる**（「あとで」「今日は休む」で即引き下がる）こと。
3. **就寝前の軽い振り返り**: 23:30頃、今夜の実績を短く返す（責めない・急かさない）。

- 締切のあるもの（原神イベント等）は上記と**別枠**で、残り日数に応じてリマインド。

### 3-3. 意図の入力方法

- **週次バックログ ＋ 夜に選ぶ** のハイブリッド:
  - 週の頭に「今週やりたいこと」を数個だけ登録（`/want` 流用を検討）。
  - 夜の号令では**ゼロから考えさせず、バックログから「今夜どれ触る」を選ぶ/確認する**軽い対話にする。
- **締切型**（原神イベント等、期限あり）と**育成型**（創作等、継続時間の確保）を分けて扱う:
  - 締切型 → カレンダー等で期限管理し、残り日数でリマインド。
  - 育成型 → 週N時間の確保を目標に、取れたかを照合。

### 3-4. トーン

- **やんわり見守り**。命令でなく提案（「〜したら？」）。1回の発話につき指摘は1つ、短文（1〜2文）。
- 明確なトリガー（逸脱・締切接近）がなければ**沈黙**（通知を出さない）。
- 「今日は休む」を常に尊重し、その日は静かに引き下がる。

### 3-5. 通知・対話チャネル

- **既存の Telegram bot** を使う。
- 号令→OWNER の返信→システムの応答、という**双方向の対話**が必要。
  （一方通行通知でなく、返信を受け取って Claude に戻す往復が要る点に留意）

## 4. 機能要件

- **F1 収集**: Windows PC と Android の「今何をしているか」を記録する（アクティブアプリ/ウィンドウ、使用時間、離席状態）。
- **F2 集約**: 各デバイスの活動記録を Linux 機に集め、時系列で蓄積・照会できるようにする。
- **F3 要約**: 夜ブロックの直近活動を要約する（上位アプリ、連続作業/滞留、意図との差分、締切の残り日数）。
- **F4 生成**: `claude -p` で、§3-2 のリズムと §3-4 のトーンに沿った伴走メッセージを生成する。
- **F5 通知/対話**: Telegram bot で送受信し、双方向の短い対話を成立させる。
- **F6 意図管理**: 週次バックログと締切アイテムを保持し、要約・生成に供給する。
- **F7 頻度制御**: 号令は1日1回、中間チェックは多くて1回。同種の指摘の連発を抑えるレート制限を持つ。

## 5. 非機能要件

- **N1 プライバシー**: ウィンドウ/アプリのタイトルには機微情報（文書名・URL 等）が混じる。
  `claude -p` に渡す前に除外・マスクするフィルタを設ける。
  ※ Linux への蓄積まではローカル完結だが、**生成の段で要約がクラウド（Claude API）に出る**点を設計に明示すること。
- **N2 低圧性**: §1・§3-4 の「超低圧・即引き下がる」を全機能に優先する上位制約とする。
- **N3 堅牢性**: 収集元デバイスがオフライン/未起動でも破綻しない（取れた分で動く）。
- **N4 セキュリティ**: デバイス間通信は信頼済みネットワーク（Tailscale 等）に限定。Telegram token / chat_id は秘密として扱う（git 外）。

## 6. 前提環境（Linux 機で実在確認すること）

- 常時起動 Linux 機（ytcheck 移行と同一機体）。claude 認証済み・Tailscale 接続済みを想定。
- ytcheck が同機で稼働（Linux + 定時実行 + `claude -p` + Telegram/Tailscale の同型先行例）。**構成の流用元として参照**。
- 既存の Telegram bot（token / chat_id）。

## 7. Linux 機が調査・判断する領域（このドキュメントで固定しない）

以下は**要件を満たす手段の選択**であり、Linux 機が既存環境を調査した上で決めること:

- **活動記録の取得手段**: ActivityWatch（PC 版 aw-watcher、Android 版 aw-android、REST API）を第一候補とするが、Android 側の取得可否（UsageStats 権限、システム更新で権限が外れる既知問題への対処含む）を実地で確認して最終決定する。
- **集約方式**: Linux から各デバイスの API を pull する案が候補。sync 方式との比較、aw-server の待受バインド変更（既定 localhost → Tailscale 到達可）の要否も含めて判断。
- **蓄積形式**: 活動ログの保存先（SQLite 等）と正規化スキーマ。
- **双方向対話の実装**: Telegram の受信（webhook / long polling）と Claude への往復配管の作り方。
- **スケジューリング**: 号令・中間チェック・就寝前・週次仕込みの定時実行手段（systemd timer / cron 等）。
- **意図管理の実体**: `/want`・カレンダー（Google Calendar 等）・独自ストアのどれで持つか。
- **プライバシーフィルタ**の具体ルール。
- **既存 ytcheck 構成のうち再利用できる部品**の洗い出し。

## 8. 進め方（Linux 機の Claude Code への指示）

1. **まず §6 の前提環境を実在確認**する（Linux 機の状態、Telegram bot、Tailscale、ytcheck 構成）。
2. §7 の論点を調査し、**要件（§1〜§5）を満たす実装計画・タスク分解を作成**する。
3. 計画を **OWNER に提示して承認を得てから実装に入る**（要件段階の想定を実装で崩さない）。
4. 段階導入を推奨: **Phase 1 = PC のみで夜ブロックの一周（号令→観測→就寝前→Telegram）を成立**させ、その後 **Phase 2 = Android 追加**、**Phase 3 = 中間チェック/締切リマインド/プライバシーフィルタ/頻度調整** の順で広げる。
5. 動作確認（実際に夜ブロックを1回通す）で検証してから各 Phase を完了とする。
